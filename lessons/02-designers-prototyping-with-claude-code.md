# Helping Designers Prototype with Claude Code

> Status: Scaffold — early experiment, still forming the lesson

## The Setup

- I'm a PM who started using Claude Code for product work (see: [lesson 01](01-building-demos-with-claude-code.md))
- Natural next question: could designers use this too?
- Not to replace Figma or design tools — but for fast directional prototyping when a full design sprint is overkill

**What I'm still figuring out:**
- Is this a tool problem, a workflow problem, or a mindset problem?
- Am I projecting my own enthusiasm onto designers who didn't ask for this?

## Why I Think This Matters

- _Expand on: the gap between "PM has an idea" and "designer has time to explore it" — Claude Code could compress that gap_
- _Expand on: designers thinking in interactions and flows, not just static screens — HTML prototypes let them test ideas faster than Figma prototypes in some cases_
- _Counter-argument: Figma is already fast. Why add another tool?_

## Early Observations

### The prompt is the spec
- _Expand on: for designers, writing a Claude Code prompt is basically writing a design brief — but one that executes immediately_
- _Does this change how designers think about specifications?_

**Think through:** Have I actually seen a designer write a prompt? What happened? If not, what am I basing this on?

### The fear of "code"
- _Expand on: Claude Code outputs HTML/CSS/JS, which can feel intimidating even though the designer never touches it_
- _How do you frame it so designers see the output, not the implementation?_

**Think through:** Is this a real barrier I've observed, or am I assuming it based on stereotypes about designers?

### What designers need that's different from what PMs need
