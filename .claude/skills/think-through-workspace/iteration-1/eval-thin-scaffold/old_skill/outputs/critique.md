# Think-Through Critique: Lesson 02 — Helping Designers Prototype with Claude Code

---

## 1. Observation Check

This lesson is almost entirely hypothetical. That's not a criticism — the scaffold itself acknowledges this — but it needs to be said plainly: **there are zero grounded observations in this file.**

Specific flags:

- **"Natural next question: could designers use this too?"** — Whose natural next question? Yours? Did a designer ask you this, or did you generate this question yourself after your own experience? There's a big difference between a pull (designer curiosity) and a push (PM enthusiasm looking for a broader audience).

- **"designers thinking in interactions and flows, not just static screens"** — This reads like a generalization about designers, not something you observed a specific designer do or say. Which designer? What interaction were they thinking about?

- **"The prompt is the spec" / "for designers, writing a Claude Code prompt is basically writing a design brief"** — This is an analogy you constructed. Has a designer actually experienced this? The scaffold's own "Think through" prompt asks this, and it's unanswered.

- **"The fear of 'code'"** — The scaffold's own "Think through" prompt asks whether this is observed or assumed. The fact that the prompt exists but no answer is provided tells me this is still assumption territory.

- **"PMs want directional answers fast. Designers want craft control."** — A clean dichotomy that sounds right but may be too neat. Have you confirmed this with designers at Buy&Ship, or is this pattern-matched from general PM/designer stereotypes?

**Bottom line:** Every claim in this scaffold traces back to your mental model of what designers might want, not to what any specific designer said or did. The scaffold is honest about this — but the critique has to name it clearly: you don't have evidence yet.

---

## 2. Lesson Depth Check

**Current framing:** "Claude Code could help designers prototype faster."

This is a tool-adoption pitch, not a lesson. It's the obvious first-pass framing: "I found a tool useful, maybe others will too."

**What's underneath it — two alternative framings:**

1. **"PMs who build with AI are going to create a new handoff problem."** If PMs start generating working prototypes, designers might feel bypassed rather than empowered. The lesson might not be "designers should use Claude Code" — it might be "PMs using Claude Code need to think carefully about how this changes the PM-designer relationship." That's a harder, more honest lesson about power dynamics in product teams.

2. **"The real test of any 'everyone can code' tool is whether a non-evangelist uses it twice."** You already have this signal buried in the "Open Questions" section ("a designer voluntarily uses Claude Code for a second time"). That's a sharper lesson than "designers could prototype with this." The lesson becomes about adoption signals vs. enthusiasm, and it applies far beyond Claude Code.

**Uncomfortable truth the lesson might be dancing around:** You might be unconsciously building a case for why designers should adopt your workflow, rather than genuinely asking whether they should. The scaffold's self-aware question — "Am I projecting my own enthusiasm onto designers who didn't ask for this?" — is the most important line in the entire file. If the answer is yes, the lesson isn't about designers at all. It's about the PM temptation to evangelize your tools.

## 3. Gap Check

### Unanswered "Think through" prompts

Every single "Think through" prompt in the scaffold is unanswered. That's five prompts, all still open:

1. What specific moment or friction made me think "a designer should be able to do this"?
2. Am I writing about what I've done, or what I imagine doing?
3. Have I actually seen a designer write a prompt?
4. Is the "fear of code" a real barrier I've observed?
5. Have I talked to designers about this? What did they actually say?

These aren't optional decoration — they're the work. Until at least 3-4 of these are answered with specifics, this scaffold can't progress.

### Missing sections / gaps

- **"What I've Tried So Far"** has no content under either sub-heading. The "actually happened" list and the "still hypothetical" list are both empty placeholders.
- **No specific designer is mentioned anywhere.** Not by name, not by role, not by reaction. The lesson is about designers but contains zero designer voices.
- **No failure mode is explored.** The open questions gesture at "when does this create more work," but there's no honest reckoning with the possibility that this just doesn't work for designers and that's fine.
- **The "PM generates and designer reacts" model** is mentioned as an open question but deserves more weight — because that's probably what's actually happening right now, and it's worth examining honestly rather than treating as a lesser option.

### Counterarguments not acknowledged

- Figma's own AI features (and other tools like Framer, Vercel v0) may already be solving this problem in ways that feel native to designers. Why Claude Code specifically?
- Designers who want to prototype in code already have tools (Webflow, Framer, even just CodePen). What's the delta?
- Adding a tool that outputs code may actually increase the designer's sense of dependency on engineering or technical PMs, which is the opposite of empowerment.

## 4. Audience Check

If cross-posted to Substack today: **it wouldn't work.** Not because the topic is bad — it's a genuinely interesting question — but because there's no story yet. There's no moment, no reaction, no result to anchor it.

- A PM who's never used Claude Code would read this and think: "Okay, so... did it work?" There's no answer.
- The scaffold assumes the reader already cares about Claude Code. A cold reader needs the "why should I care" hook, which has to come from a real moment, not from a logical argument about workflow compression.
- **The one shareable line doesn't exist yet.** The closest candidate is the signal framework in the "Lesson (Not Ready Yet)" section — "Signal that this doesn't work: I'm the only one generating and designers are just reviewing my output" — that's a good line, but it needs a real result behind it to land.

For the eventual article, the hook is almost certainly going to be a specific designer's reaction — positive or negative. The scaffold already knows this (see the "If Cross-Posting to Medium" section). The work is to go get that reaction.

---

## 5. Verdict

**Not ready.** This is a well-structured scaffold for an experiment that hasn't happened yet. The self-awareness is strong — the "Think through" prompts ask exactly the right questions, and the "What I'm still figuring out" framing is honest. But the scaffold is doing the work of knowing what to ask without doing the work of answering.

The next step isn't writing — it's doing the experiment and filling in the "Think through" prompts with real specifics.

---

## 3 Questions to Sit With

1. **Have you actually shown a Claude Code prototype to a designer at Buy&Ship yet?** If not, what's stopping you — and is the blocker itself part of the lesson? (e.g., you might be avoiding it because you suspect the reaction won't match your enthusiasm.)

2. **If you removed the word "designer" from this lesson and replaced it with "non-technical teammate," would the lesson change at all?** If not, you might be writing a lesson about AI tool adoption more broadly, and "designers" is just the example you happened to pick because of your own workflow. That's a different — and possibly more honest — lesson.

3. **What if the real lesson turns out to be: "I tried to get designers to use Claude Code and they didn't want to"?** Would you still write it? That might be the more valuable and more shareable lesson — the gap between PM enthusiasm and team adoption. Decide now whether you're open to that outcome.
